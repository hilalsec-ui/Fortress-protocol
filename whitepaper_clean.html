01The Vision & Problem

The Problem with
Traditional Lotteries

The global lottery industry moves trillions in ticket sales annually — yet the mechanism by which winners are selected is almost universally a black box. Players deposit funds into an opaque server. A private algorithm draws a result. No participant can verify the process was fair.

A single party controls prize selection with no external audit trail.

05Provably Fair Engine

Switchboard V3 TEE
On-Chain Randomness

SB On-Demand Program (mainnet): Aio4gaXjXzJNVLtzwtNVmSqGKpANtXhybbkhtAC94ji2

Instead of a private RNG, every lottery type in Fortress Protocol uses Switchboard V3 On-Demand VRF — a Trusted Execution Environment oracle running inside a hardware-secured Intel SGX enclave. The enclave generates a 32-byte verifiable random value that even the oracle operator cannot bias, inspect, or predict.

Each draw is a two-step commit-reveal protocol:

  • sb_value[0..31] — the 32-byte VRF output written by the SGX enclave after commit
  • reveal_slot — Solana slot when the oracle revealed; must be strictly greater than request slot (anti-replay)
  • user_commitment — a 32-byte user-supplied commitment XOR'd in at fulfillment; neither party can bias outcome alone
// Winner derivation (oracle.rs — create_lottery_entropy_from_slot)
s0..s3 = sb_value split into four u64s (Xoshiro256++)
a0 = s0 XOR s2    a1 = s1 XOR s3
final = a0 XOR a1 XOR (commitment ⊕ vault_seed)
winner = final % participant_count

The vault_seed encodes the lottery type, tier number, and round number — guaranteeing that two draws at the exact same slot still produce different winners. If reveal_slot == 0, or the revealed slot is not strictly greater than the request slot, fulfill_draw_entropy rejects with DrawNotYetReady.

06Draw Trigger System

How Draws Get Triggered

Fortress Protocol supports two trigger methods that work together as a resilient dual-path system.

[Bot] AUTO-TRIGGER (ON-HOUSE)

Crank Bot — Fully Automated

The protocol runs a keeper bot (crank) that monitors all 16 tiers continuously. The moment a tier becomes eligible for a draw, the crank automatically initiates the full draw sequence with zero human intervention.

DetectionPolls on-chain vault state every few seconds
Step 1Creates Switchboard randomness account, submits request_draw_entropy + commit in one transaction
Oracle WaitWaits ~2.5 seconds for SGX enclave oracle to produce verifiable random value
Step 2Submits reveal transaction — derives winner index and pays out 95% of the pool instantly
CostAll Switchboard oracle fees, rent, and gas paid by protocol treasury — users pay nothing
crank detects → commit → oracle TEE → reveal → winner paid
[Wrench] FALLBACK TRIGGER (COMMUNITY)

Automatic-First — Community Fallback

If automation is temporarily unavailable, any connected wallet can step in. This 3-click fallback flow ensures the protocol is fully permissionless: no single point of failure.

Step 1Click Request Draw — wallet signs a tx creating an ephemeral SB randomness account, commits randomness, creates PendingDraw PDA on-chain
Oracle WaitSGX enclave oracle processes the commitment (~5-10 seconds)
Step 2Click Oracle Reveal — wallet signs the reveal transaction confirming the oracle output on-chain
Step 3Click Finalize Draw — program verifies oracle output, derives winner, pays prize pool in same block
CostTreasury reimburses all SB oracle fees — settler pays only standard Solana gas (~0.000005 SOL)
BountySettler earns an FPT bounty from the treasury for finalizing the draw
user fallback step 1 → oracle TEE → step 2 → step 3 → winner paid + FPT bounty
